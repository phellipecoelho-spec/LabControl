    - Traits: HasFactory, HasUuids, SoftDeletes, LogsActivity
    - $table = 'inventory_items'
    - $fillable: name, code, description, category_id, supplier_id, unit, min_stock, batch_lot, expiry_date, physical_location, user_id, created_by, updated_by, deleted_by
    - $casts: expiry_date => 'date', min_stock => 'integer'
    - $appends: ['current_balance', 'is_critical']
    - Relationships:
      - category() → belongsTo(InventoryCategory::class)
      - supplier() → belongsTo(Supplier::class)
      - user() → belongsTo(User::class)
      - movements() → hasMany(InventoryMovement::class)
      - deletedBy() → belongsTo(User::class, 'deleted_by')->withDefault()
    - Computed accessors:
      - getCurrentBalanceAttribute(): int — reads `$this->movements()->latest('created_at')->value('balance_after') ?? 0` (per D-10, balance computed from movements)
      - getIsCriticalAttribute(): bool — returns `$this->current_balance <= $this->min_stock` (per D-12)
    - Scopes: search (name/code), byCategory, bySupplier, byUnit, critical (whereRelation to filter items whose balance from movements <= min_stock — this scope may need a subquery for the latest balance_after per item)

    **C. `InventoryMovement.php` (Model)**
    - Traits: HasFactory, HasUuids (NO SoftDeletes — movements are immutable per D-06 research)
    - $table = 'inventory_movements'
    - $fillable: item_id, type, quantity, balance_after, reason, notes, user_id, created_by
    - $casts: quantity => 'integer', balance_after => 'integer'
    - Relationships:
      - item() → belongsTo(InventoryItem::class)
      - user() → belongsTo(User::class)
    - Scope: byType, byItem, byUser, byDateRange

    **D. `InventoryMovementService.php` (Service)**
    - Single public method: `recordMovement(array $data): InventoryMovement`
    - Implement the THREE-LAYER negative stock defense per CONTEXT.md risks:
      1. **DB transaction**: Wrap everything in `DB::transaction(function () use ($data) { ... })`
      2. **Row-level lock**: `$item = InventoryItem::where('id', $data['item_id'])->lockForUpdate()->first()` — prevents concurrent read-check-write race conditions
      3. **Application validation**: Compute direction based on type (in_array($type, ['purchase', 'return']) ? '+' : '-'). If direction is decrementing AND new balance < 0, throw `InsufficientStockException` (create this as a custom exception class in `app/Exceptions/InsufficientStockException.php`) with message including available and requested quantities
    - Set `$data['balance_after'] = $newBalance` and `$data['user_id'] = auth()->id()`
    - Return `InventoryMovement::create($data)`
    - Note: The service does NOT check min_stock or return is_critical — that is the controller/API layer's responsibility (separation of concerns). The service owns data integrity (balance, locking, audit)
