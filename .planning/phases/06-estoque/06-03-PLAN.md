    - `getById(id)` → GET /inventory-movements/{id}

    **E. `store/InventoryItemStore.ts`** — Pinia composition store following EquipmentStore pattern:
    - State: items (ref<InventoryItem[]>), loading, pagination (current_page, last_page, total, per_page)
    - State: categories (ref<InventoryCategory[]>), suppliers (ref<Supplier[]>)
    - Actions:
      - `fetchAll(params?)` — calls service.list, handles both array and paginated response (same pattern as EquipmentStore)
      - `fetchById(id)` — returns single item data
      - `create(data)` — POST, returns created item
      - `update(id, data)` — PUT, returns updated item
      - `destroy(id)` — DELETE, removes from items array
      - `fetchCategories()` — GET inventory-categories
      - `fetchSuppliers()` — GET /suppliers (reuses existing Supplier endpoint from Phase 5)
      - `fetchItemMovements(itemId, params?)` — calls service.getMovements

    - State: lastCreatedCritical (ref<boolean>) — flag for toast alerting

      - `fetchAll(params?)` — calls movementService.list with filters
      - `create(data)` — calls movementService.create; checks response.meta.is_critical to set lastCreatedCritical
      - `getById(id)` — returns single movement
    - Use InvenotryMovementService for API calls

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User browser → Vue app | XSS vector: item names, descriptions, reasons rendered in DataTable and detail pages |
| Vue app → API (Axios) | Sanctum token in Authorization header; CSRF via SameSite cookie |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-08 | Tampering | DataTable dynamic content rendering | medium | mitigate | Vue 3 template escaping (mustache syntax {{ }} is auto-escaped; v-html never used in any inventory component) |
