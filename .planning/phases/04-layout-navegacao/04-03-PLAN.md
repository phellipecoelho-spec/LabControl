<objective>
Complete the app shell with three finishing layers: (1) permission-based module filtering so users only see modules they have access to, (2) mobile responsive drawer behavior so the sidebar becomes an overlay on viewports <768px, and (3) accessibility polish (ARIA labels, skip-to-content, keyboard navigation). Plan 03 finalizes the navigation UX for all device sizes and permission levels.

Purpose: After Plans 01 and 02 built the foundation and the visible components, this plan adds the two critical runtime behaviors that make the shell production-ready — dynamic visibility based on user permissions (D-17) and responsive adaptation for mobile devices (D-03). Without this plan, all modules would always be visible and the sidebar would not work on phones.

Output: Updated AppSidebar, AppLayout, and AppTopbar with permission filtering, mobile drawer, and accessibility enhancements.
</objective>

  <action>
    Update `AppSidebar.vue` to implement permission-based module filtering:

    **1. Add a `filteredPanelMenuModel` computed (per RESEARCH.md Pitfall 5 fix):**
    Replace or wrap the existing `panelMenuModel` with permission filtering logic. The model must be a `computed` that depends on `authStore.user` so it updates reactively when user data loads:
    ```typescript
    const filteredPanelMenuModel = computed(() => {
      // Transform navigationTree into PanelMenu model
      // Filter each category's items by permission/role
      return navigationTree
        .filter((item): item is NavCategory => 'items' in item && item.key !== undefined)
        .map(category => ({
          key: category.key,
          label: category.label,
          icon: category.icon,
          items: category.items.filter(module => {
            // If module has permission, check hasPermission
            if (module.permission) {
              return authStore.hasPermission(module.permission)
            }
            // If module has roles, check hasRole (any match)
            if (module.roles && module.roles.length > 0) {
              return module.roles.some(role => authStore.hasRole(role))
            }
            // No permission check needed — always visible
            return true
          })
        }))
        .filter(category => category.items.length > 0) // Hide empty categories
    })
    ```

    **2. Filtering rules (per D-17 and UI-SPEC Navigation Structure):**
    - `Equipamentos`: requires `'equipamentos.view'` permission
    - `Estoque`: requires `'estoque.view'` permission
    - `Movimentações`: requires `'movimentacoes.view'` permission
    - `Empréstimos`: requires `'emprestimos.view'` permission
    - `Calibrações`: requires `'calibracoes.view'` permission
    - `Aferições`: requires `'afericoes.view'` permission
    - `Manutenções`: requires `'manutencoes.view'` permission
    - `Usuários`: requires role `'admin'` or `'supervisor'`
    - `Perfis de Acesso`: requires role `'admin'`
    - `Logs de Auditoria`: requires role `'admin'` or `'auditor'`
    - `Relatórios`: requires `'relatorios.view'` permission
    - `Dashboard`: always visible (no permission check)

    **3. Empty category handling:**
    If all items in a category are filtered out, the category heading must also be hidden. The `.filter(category => category.items.length > 0)` step above handles this.

    **4. Use `filteredPanelMenuModel` in the template** instead of the unfiltered model. Replace `panelMenuModel` with `filteredPanelMenuModel` in the `<PanelMenu :model="...">` binding.

    **5. Keep the Dashboard link always visible** — it is rendered outside the PanelMenu and should never be filtered.

    **6. Security note** (per RESEARCH.md Security Domain): Sidebar filtering is UX optimization only. Backend route guards already enforce authorization. The filtering prevents users from seeing links they cannot access, but does not prevent direct URL navigation. Do NOT add security comments in this component — it is intentionally a UI-only concern.

    **7. Do NOT add loading states** for permission filtering. The `authStore.user` data is already loaded by the router `beforeEach` guard before any authenticated page renders. If `authStore.user` is null, all modules with permission checks will be hidden (Dashboard remains visible).

    Update the PanelMenu `:model` binding to use the new computed. Remove any old unfiltered model references.
  </action>

    ### A) AppSidebar.vue — Accessibility enhancements

    1. **ARIA labels on icon-only sidebar state:**
       When collapsed (icon-only), each `router-link` and PanelMenu item must have `aria-label` matching the module's label. Since PanelMenu doesn't natively support per-item aria-label, add a `v-tooltip.right` directive (PrimeVue Tooltip) on the collapsed sidebar items to show module names on hover:

       ```vue
       <router-link
         to="/"
         class="app-sidebar__dashboard"
         :class="{ 'app-sidebar__dashboard--active': isDashboardActive }"
         v-tooltip.right="collapsed ? 'Dashboard' : undefined"
         aria-label="Dashboard"
       >
       ```

       Note: PanelMenu items can't easily receive v-tooltip per-item. Skip tooltip on PanelMenu items in scope-restricted manner — the CSS `display:none` on labels means they're not focusable when collapsed, so tooltips are a nice-to-have, not a blocker.
